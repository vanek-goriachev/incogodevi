# Требования. Go Dependencies Visualizer

> Фаза 1 из 5. Документ фиксирует функциональные и нефункциональные требования к инструменту интерактивной визуализации зависимостей Go-кода. Архитектура и декомпозиция — в следующих фазах.

## 1. Обзор

**Что.** Веб-инструмент для статического анализа Go-проектов и интерактивной визуализации графа зависимостей между сущностями (пакеты, структуры, интерфейсы, функции, методы, поля, переменные, константы). Позволяет задавать точки входа, выделять неиспользуемый (мёртвый) код, настраивать уровень детализации графа.

**Для кого.** Go-разработчики и тимлиды, работающие с legacy-кодом и крупными проектами; учебные заведения на курсах по Go; QA/архитекторы.

**Зачем.** Ускорение рефакторинга за счёт безопасного удаления мёртвого кода, сокращение времени онбординга (визуализация структуры), ускорение CI за счёт исключения мёртвых участков из тестирования.

**Выигрыш vs аналоги.** В отличие от `staticcheck` (только текстовый `U1000`, ограничен рамками пакета) и `go-callvis` (статическая картинка без интерактива, без мёртвого кода, без множественных точек входа) программа совмещает: глубокий анализ через типовую систему (`go/types.Implements`, учёт embedding и aliases), интерактивный граф в браузере (zoom/pan/drag/скрытие ветвей/агрегация), «лес» пользовательских точек входа и экспорт (PNG/SVG/TXT/JSON).

## 2. Функциональные требования

Формат: `**FR-XX.** <формулировка>. — Источник: [ссылка]. — Acceptance: <критерий>`.

### 2.1 Анализ кода

**FR-01.** Приём Go-проекта через веб-интерфейс в виде ZIP-архива (upload или drag-n-drop).
- Источник: Уточнено с пользователем 2026-04-18; [ТЗ, sections/04-requirements.typ, «Требования к организации входных данных»]
- Acceptance: пользователь перетаскивает `.zip` в зону загрузки или выбирает его через диалог — архив принимается сервером с HTTP 200; при отсутствии `go.mod` после распаковки — HTTP 400 с сообщением «valid Go module not found».

**FR-02.** Безопасная распаковка ZIP во временный каталог сервера с проверкой на zip-slip.
- Источник: Best practice
- Acceptance: попытка загрузить архив с путями, выходящими за пределы извлечения (`../`, абсолютные пути), отклоняется до записи на диск; временный каталог удаляется после завершения обработки запроса.

**FR-03.** Рекурсивный обход всех `.go`-файлов проекта с учётом `go.mod`.
- Источник: [ТЗ, sections/04-requirements.typ, «Состав выполняемых функций», п. 1]
- Acceptance: для проекта с `N` Go-файлами в результат анализа попадают все `N` (кроме явно исключённых `_test.go`, если выключены в фильтре).

**FR-04.** Загрузка пакетов через `golang.org/x/tools/go/packages` в режиме с типовой информацией (`NeedTypes | NeedTypesInfo | NeedSyntax | NeedImports | NeedDeps | NeedModule`).
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к информационным структурам и методам решения»]; research-notes §4
- Acceptance: `packages.Load` возвращает `*types.Package` и непустой `*types.Info.Uses` для тестового проекта из `testdata/`; ошибки импорта логируются, но не прерывают анализ (см. NFR-08).

**FR-05.** Извлечение сущностей: пакетов, импортов, структур, интерфейсов, функций, методов, переменных, констант, полей структур.
- Источник: [ТЗ, sections/04-requirements.typ, «Состав выполняемых функций», п. 1]
- Acceptance: для контрольного проекта с заранее посчитанным числом каждой категории анализатор выдаёт совпадающий count (±0).

### 2.2 Построение графа зависимостей

**FR-06.** Автоопределение точек входа: **все** `func main()` в пакетах `package main` внутри проекта.
- Источник: Уточнено с пользователем 2026-04-18; [ТЗ, sections/03-purpose.typ, «Функциональное назначение», п. 2]
- Acceptance: для monorepo с `cmd/a/main.go` и `cmd/b/main.go` по умолчанию оба попадают в список entry points и отображаются в UI.

**FR-07.** Пользовательское задание дополнительных точек входа через UI (выбор конкретной функции, метода или типа из списка всех сущностей).
- Источник: [ТЗ, sections/04-requirements.typ, «Состав выполняемых функций», п. 2]
- Acceptance: после добавления пользователем сущности `pkg/api.Handler` в entry points и перестройки графа — её узел становится корневым и достижимый из него подграф помечается как живой.

**FR-08.** Построение направленного графа зависимостей от множества точек входа (обход use-def графа).
- Источник: [ТЗ, sections/04-requirements.typ, «Состав выполняемых функций», п. 2]
- Acceptance: для тестового проекта с 3 функциями, где `A→B→C`, при entry = {A} в графе 3 узла и 2 ребра; при entry = {B} — 2 узла и 1 ребро.

**FR-09.** Анализ зависимостей через интерфейсы с использованием `go/types.Implements` (учёт embedding, type aliases, методов через встроенные структуры).
- Источник: [ТЗ, sections/04-requirements.typ, «Состав выполняемых функций», п. 2, «корректная обработка зависимостей через интерфейсы»]; Уточнено с пользователем 2026-04-18
- Acceptance: для проекта с интерфейсом `I` и двумя реализациями `A`, `B`, где только `A` используется в entry points, `B.Method()` помечается как мёртвый; embedded-имплементации (через `struct { A }`) корректно детектируются.

**FR-10.** Построение «леса» зависимостей от нескольких entry points одновременно.
- Источник: [ТЗ, sections/03-purpose.typ, п. 5]
- Acceptance: при entry = {A, B}, где A и B не связаны, граф содержит 2 компоненты связности; сущности, не достижимые ни из одного корня, помечаются как мёртвые.

### 2.3 Визуализация и интерактивность

**FR-11.** Отрисовка графа в SPA в виде узлов и рёбер с использованием Cytoscape.js.
- Источник: Уточнено с пользователем 2026-04-18; [ТЗ, sections/04-requirements.typ, «Требования к интерфейсу»]
- Acceptance: после загрузки проекта в течение ≤ NFR-02 (5 с) граф виден в главной области интерфейса.

**FR-12.** Zoom и панорамирование области графа мышью/тачпадом.
- Источник: [ТЗ, sections/04-requirements.typ, «Специальные требования», п. 2]
- Acceptance: колесо мыши масштабирует граф; зажатое перетаскивание пустой области — панорамирует.

**FR-13.** Перетаскивание (drag) отдельных узлов мышью.
- Источник: [ТЗ, sections/04-requirements.typ, «Интерактивная визуализация», п. 2]
- Acceptance: пользователь перемещает узел в новое положение, связанные рёбра пересчитываются; позиция сохраняется до перезагрузки.

**FR-14.** Фильтры детализации: скрытие/отображение узлов по типу сущности (переменные, константы, поля, функции, методы, структуры, интерфейсы, пакеты).
- Источник: [ТЗ, sections/04-requirements.typ, «Интерактивная визуализация», п. 3]
- Acceptance: в панели фильтров есть 8 тумблеров; выключение любого приводит к мгновенному исчезновению соответствующих узлов без повторного запроса к серверу.

**FR-15.** Визуальное выделение мёртвых узлов (недостижимых из entry points) через отдельный стиль (например, серый + пунктирная обводка).
- Источник: [ТЗ, sections/04-requirements.typ, «Интерактивная визуализация», п. 4]
- Acceptance: в тестовом графе все узлы, заранее помеченные как мёртвые, имеют CSS-класс `dead` и визуально отличимы от живых.

**FR-16.** Временное скрытие (collapse) отдельных ветвей по клику на узел.
- Источник: [ТЗ, sections/04-requirements.typ, «Интерактивная визуализация», п. 5]
- Acceptance: контекстное меню/кнопка «hide subtree» скрывает выбранный узел и всех его потомков; повторное действие возвращает их.

**FR-17.** Отображение метаинформации узла (тип сущности, файл, строка, сигнатура) при hover или click.
- Источник: [ТЗ, sections/04-requirements.typ, «Интерактивная визуализация», п. 6]
- Acceptance: при наведении на узел в течение ≥ 300 мс появляется tooltip с полями `kind`, `name`, `package`, `file:line`; при клике — информация выводится в боковой панели.

**FR-18.** Автоматическая агрегация подграфов в «коллапсированные» узлы при объёме графа > 1000 узлов.
- Источник: [ТЗ, sections/04-requirements.typ, «Специальные требования», п. 3]
- Acceptance: для проекта с >1000 сущностями граф изначально отображается с агрегацией по пакетам (один узел = один пакет с индикатором размера); раскрытие узла заменяет его на детальный подграф.

### 2.4 Обнаружение мёртвого кода

**FR-19.** Пометка всех сущностей, недостижимых из выбранных точек входа, как мёртвых в модели графа.
- Источник: [ТЗ, sections/03-purpose.typ, п. 2]; [sections/04-requirements.typ, «Состав выполняемых функций», п. 3]
- Acceptance: для тестового проекта с заранее известным списком недостижимых сущностей результат совпадает (100% precision и recall на `testdata/`).

**FR-20.** Формирование текстового списка мёртвых сущностей с указанием имени, типа, файла и номера строки.
- Источник: [ТЗ, sections/04-requirements.typ, «Экспорт и отчетность», п. 1]
- Acceptance: список отображается в отдельной панели UI и имеет формат `<kind> <pkg>.<name> — <file>:<line>`; пустой список показывает сообщение «no dead code detected».

### 2.5 Экспорт и отчёты

**FR-21.** Экспорт текущего вида графа в PNG.
- Источник: [ТЗ, sections/04-requirements.typ, «Экспорт и отчетность», п. 2]
- Acceptance: кнопка «Export PNG» инициирует скачивание PNG-файла с именем `<project>-graph-<timestamp>.png`, изображение соответствует видимому вьюпорту.

**FR-22.** Экспорт текущего вида графа в SVG.
- Источник: [ТЗ, sections/04-requirements.typ, «Экспорт и отчетность», п. 2]
- Acceptance: кнопка «Export SVG» отдаёт валидный SVG, открывающийся в браузере и редакторах векторной графики.

**FR-23.** Экспорт отчёта о мёртвом коде в TXT.
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к организации выходных данных», п. 2]
- Acceptance: содержимое файла совпадает построчно с FR-20 (UTF-8, LF, без BOM).

**FR-24.** Экспорт отчёта о мёртвом коде в JSON с типизированной схемой (`{entries: [{kind, package, name, file, line}]}`).
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к организации выходных данных», п. 2]
- Acceptance: файл парсится стандартным `JSON.parse`; схема описана в `docs/` и покрыта unit-тестом маршалинга на сервере.

### 2.6 Конфигурация и точки входа

**FR-25.** Панель UI для управления entry points, фильтрами и параметрами визуализации.
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к интерфейсу», п. 2]; Уточнено с пользователем 2026-04-18 (конфиг-файл JSON/YAML вне scope, только UI)
- Acceptance: панель содержит: список текущих entry points с возможностью add/remove, тумблеры фильтров из FR-14, селектор цветовой темы и размера узла.

**FR-26.** Сохранение пользовательских настроек (entry points, фильтры, раскладка) в `localStorage` браузера.
- Источник: Уточнено с пользователем 2026-04-18
- Acceptance: после перезагрузки страницы для **того же** проекта восстанавливаются последние entry points, состояния фильтров и ручные позиции узлов; при загрузке **другого** проекта — применяются дефолты.

## 3. Нефункциональные требования

### 3.1 Производительность

**NFR-01.** Полный анализ проекта ≤ 30 с для кодовой базы до 50 000 LOC.
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к временным характеристикам», п. 1]
- Acceptance: на контрольном проекте из `testdata/` (~50k LOC) `go test -bench` измеряет время от приёма архива до отдачи JSON-графа ≤ 30 с на машине из тех-требований (см. NFR-07).

**NFR-02.** Время первой отрисовки графа ≤ 5 с после получения данных.
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к временным характеристикам», п. 2]
- Acceptance: Playwright-тест фиксирует интервал между получением JSON на клиенте и событием `cy.ready()` ≤ 5000 мс для графа с ≤ 1000 узлов.

**NFR-03.** Отклик UI (drag, hide, filter-toggle) < 100 мс.
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к временным характеристикам», п. 3]
- Acceptance: профилирование в Chrome DevTools показывает < 100 мс от user gesture до первого frame обновления для графа ≤ 1000 узлов.

**NFR-04.** Лимиты загружаемого архива: ≤ 50 МБ raw, ≤ 50 000 LOC, ≤ 10 000 файлов.
- Источник: Уточнено с пользователем 2026-04-18; согласовано с NFR-01
- Acceptance: превышение любого лимита приводит к HTTP 413/422 с понятным сообщением на английском до начала анализа.

### 3.2 Совместимость

**NFR-05.** Поддержка Go 1.24 и выше для анализируемого кода и серверного рантайма. Серверный рантайм собирается на Go 1.26 (актуальный stable на 2026-04-19, см. https://go.dev/doc/devel/release).
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к надёжности», п. 1; «Программные средства»]; версии актуализированы 2026-04-19
- Acceptance: `go.mod` серверной части содержит `go 1.26`; CI запускает тесты на матрице Go `{1.25, 1.26}`. Исполнитель **обязан** сверить актуальную версию Go на go.dev перед фиксацией в go.mod на старте фазы реализации.

**NFR-06.** Клиент работает в последних версиях Chrome, Firefox, Edge, Safari с поддержкой ES2020 и SVG.
- Источник: [ТЗ, sections/04-requirements.typ, «Программные средства», п. 2]
- Acceptance: `browserslist` в `web/package.json` ограничивает `last 2 versions` указанных движков; Playwright прогоняет smoke-тест на Chromium + WebKit.

**NFR-07.** Сервер работает на Linux, Windows 10/11, macOS (x86-64 или ARM64); требования к железу: ≥ 2 ГБ RAM, ≥ 100 МБ диска.
- Источник: [ТЗ, sections/04-requirements.typ, «Технические средства», п. 1]
- Acceptance: Docker-образ собирается multi-arch (`linux/amd64`, `linux/arm64`); запуск на macOS и Windows проверен вручную.

### 3.3 Надёжность

**NFR-08.** При ошибке импорта отдельного пакета анализ продолжается, пользователю возвращается частичный граф и список предупреждений.
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к надёжности», п. 2]
- Acceptance: интеграционный тест с заведомо сломанным импортом получает HTTP 200, граф без сломанного пакета и массив `warnings[]` в ответе.

**NFR-09.** UI не требует перезагрузки страницы при ошибках взаимодействия с графом.
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к надёжности», п. 3]
- Acceptance: Playwright-тест вызывает серию недопустимых действий (скрыть несуществующий узел, добавить дублирующийся entry point) — ошибки ловятся Error Boundary, пользователь видит toast без потери состояния.

**NFR-10.** Восстановление после сбоя — перезагрузка страницы; `localStorage` восстанавливает настройки (FR-26).
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к надёжности», п. 4]
- Acceptance: ручной kill вкладки и повторное открытие URL восстанавливает последнее состояние настроек.

### 3.4 Удобство и UX

**NFR-11.** От загрузки ZIP до отображения графа — не более 3 действий пользователя (drag-drop → wait → граф).
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к интерфейсу», п. 3]; Best practice
- Acceptance: пользовательский сценарий проходит за 3 клика/жеста без обязательной настройки параметров.

**NFR-12.** Язык UI — английский.
- Источник: Уточнено с пользователем 2026-04-18
- Acceptance: все видимые строки UI — английские; отсутствует i18n-слой (зафиксированные строки).

### 3.5 Безопасность

**NFR-13.** Обработка загруженного проекта происходит в изолированном временном каталоге без доступа к файловой системе вне него.
- Источник: [ТЗ, sections/04-requirements.typ, «Требования к защите информации и программы», п. 2]
- Acceptance: сервер распаковывает ZIP в `os.MkdirTemp`, после завершения обработки каталог удаляется; `filepath.Clean` и проверка на `..` при распаковке.

**NFR-14.** Лимиты NFR-04 применяются до начала распаковки/анализа для защиты от DoS.
- Источник: Best practice
- Acceptance: размер запроса ограничивается `http.MaxBytesReader(…, 50MB)`; число файлов в архиве проверяется до полной распаковки.

## 4. Ограничения

- **Тех-стек (решено).** Серверная часть — Go 1.26, стандартная библиотека + `golang.org/x/tools/go/packages`; фронтенд — React 19 + TypeScript 6 + Vite 8, граф-библиотека Cytoscape.js 3.33; доставка — Docker-образ (multi-arch), HTTP-сервер встраивает статику через `embed.FS`. Версии актуальны на 2026-04-19; все значения — сверены с go.dev / react.dev / vitejs.dev / typescriptlang.org / js.cytoscape.org. **Исполнитель обязан** свериться с актуальной документацией каждой зависимости в начале фазы реализации (phase 3) и перед каждым релизом.
- **Репозиторий.** Monorepo `/server` + `/web` + `/docs` + `/tasks`.
- **Workflow.** Feature-branches + PR в `main`; Conventional Commits; GitHub Actions (lint + test, без авто-publish образа).
- **Временные.** Защита запланирована на 26.04.26 – 15.05.26 (ТЗ, `sections/07-stages.typ`); дата старта реализации сдвинута относительно плана ТЗ — см. §5.
- **Правовые.** Программа не обрабатывает конфиденциальных данных; лицензия репозитория — MIT (см. `LICENSE`).
- **Бюджетные.** Одиночный исполнитель (Горячев И. С., БПИ236).

## 5. Границы MVP

### 5.1 В scope для защиты

Все функциональные требования раздела §2 (FR-01 … FR-26) и все нефункциональные §3 (NFR-01 … NFR-14). Пользователь явно выбрал scope «Полный по ТЗ» (2026-04-18).

Ключевые MVP-буллеты:
- ZIP upload + drag-n-drop.
- Анализ через `go/packages` + `go/types.Implements` (embedding, aliases).

### 5.3 Явно вне scope

- Локализация интерфейса — только английский.
- Сохранение снапшотов анализа на сервере / база данных — вне scope (stateless сервер).

- Поддержка CGO-специфичных анализов и ассемблерных вставок.

## 6. Демо на защите

**Проекты.** Два open-source Go-проекта разного размера:
1. Небольшой (5–10k LOC) — ориентир: `urfave/cli` или `spf13/cobra` (финальный выбор фиксируется на фазе 2 после профилирования).
2. Средний (30–50k LOC) — ориентир: `go-chi/chi` или сопоставимый; должен уложиться в NFR-01.

**User journey (3–5 шагов):**
1. Открыть локально запущенный Docker-контейнер (`http://localhost:8080`).
2. Перетащить ZIP-архив маленького проекта в зону загрузки.
3. Дождаться графа (≤ 5 с), показать zoom/pan/drag и подсветку мёртвого кода.
4. Добавить вручную дополнительную entry point (например, `cmd/foo/bar.Func`), перестроить граф, показать изменение подсветки.
5. Экспортировать SVG и TXT-отчёт о мёртвом коде.
6. (При наличии времени) Загрузить средний проект, показать агрегацию при > 1000 узлов и фильтры по типам сущностей.
