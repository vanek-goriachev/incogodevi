# Архитектурные варианты

> Фаза 2. Стек зафиксирован в `requirements.md` §4: Go 1.21+ stdlib + `golang.org/x/tools/go/packages`, React 18 + TS + Vite + Cytoscape.js, Docker multi-arch, `embed.FS`. Варьируются только свободные оси — **жизненный цикл анализа** и **стратегия хранения артефактов**. Выбор граф-библиотеки, фреймворка фронтенда и формата доставки не пересматриваются.

## Общее для всех вариантов

- **Backend:** Go 1.21, только stdlib `net/http` + `embed`, анализ — `golang.org/x/tools/go/packages`, `go/types` (для `types.Implements`, embedding, aliases), `go/ast`.
- **Frontend:** React 18 + TypeScript + Vite, Cytoscape.js (+`cytoscape-svg` для FR-22), собирается отдельным stage в Dockerfile и встраивается в бинарь через `embed.FS`.
- **Delivery:** multi-stage `Dockerfile` (Node → Go → distroless/alpine), образ multi-arch (`linux/amd64`, `linux/arm64`).
- **Хранение настроек клиента:** `localStorage` (FR-26, NFR-10).
- **Безопасность ZIP:** `http.MaxBytesReader` + `filepath.Clean` + проверка `..`, распаковка в `os.MkdirTemp` (NFR-13, NFR-14).

## Вариант A — «Sync + Stateless»

### Тех-стек
- Backend: Go stdlib (`net/http`, `encoding/json`), синхронные хэндлеры.
- Frontend: React + Cytoscape.js, состояние графа живёт только в памяти вкладки.
- Build/tooling: Vite + `go build`, без очередей/воркеров.
- Delivery: одиночный бинарь + Docker-образ.

### Ключевые идеи
- Один endpoint `POST /api/analyze` принимает multipart ZIP + JSON с entry points и фильтрами → распаковывает → анализирует → отдаёт полный JSON-граф в ответе.
- Сервер полностью stateless: после возврата ответа всё временное удаляется.
- Перезапуск анализа с другими entry points = повторная загрузка ZIP (или клиент кэширует ZIP и шлёт ещё раз).
- Прогресс — только спиннер, без процентов.

### Плюсы
- Минимум кода: нет job-менеджера, нет polling'а, нет SSE, нет TTL-GC.
- Тривиальный деплой и масштабирование (любая реплика обрабатывает любой запрос).
- Легко покрыть интеграционными тестами (один запрос — один ответ).
- Память/диск не растут со временем.

### Минусы и риски
- 30 с блокирующий HTTP-запрос → риск срезки прокси/клиента таймаутом; нельзя показать прогресс (конфликт с NFR-01 по UX-восприятию, формально проходит).
- **FR-07 (итеративная смена entry points) ощущается как «каждый раз заново»** — либо пользователь перезагружает ZIP, либо мы держим ZIP на клиенте и шлём целиком при каждом пересчёте.
- Большие проекты (ближе к 50k LOC) — риск ретрая/обрыва на стороне браузера.
- Нельзя переподключиться после обрыва: результат теряется.

### Сложность освоения: **Low**

### Соответствие требованиям
- Полностью закрывает: FR-01..06, FR-08..10, FR-15, FR-19..24, все NFR по производительности (формально).
- **Потенциальный конфликт:** UX итеративного анализа (FR-07, FR-11 NFR-02 по ощущению), отсутствие частичного графа при ошибке импорта (NFR-08 требует массив `warnings[]` — работает, но без стриминга).
- Требуется CI-таймаут клиент-сервера ≥ 35 с.

## Вариант B — «Async Jobs + In-memory Project Cache» (рекомендуемый компромисс)

### Тех-стек
- Backend: Go stdlib + собственный job-менеджер на горутинах + `sync.Map`/`sync.RWMutex` поверх in-memory стораджа.
- Frontend: React + Cytoscape.js + `fetch` polling (500 мс) до статуса `ready`.
- Build/tooling: тот же Vite + `go build`.
- Delivery: одиночный бинарь + Docker.

### Ключевые идеи
- `POST /api/projects` → распаковка ZIP + разбор через `packages.Load` → возвращает `project_id`. Парсинг закэширован в памяти (TTL idle 30 мин).
- `POST /api/projects/{id}/analyze` → создаёт job, возвращает `{job_id, status: "queued"}`.
- `GET /api/projects/{id}/status` → polling-эндпоинт: `queued | running | done | failed` + прогресс по фазам.
- `GET /api/projects/{id}/graph` → последний посчитанный граф.
- Смена entry points дешёвая: AST/types уже в памяти, пересчитывается только reachability → укладываемся в <1 с.
- Warnings (NFR-08) возвращаются как поле в `status`/`graph`.

### Плюсы
- Отличный UX для FR-07: итеративный анализ — мгновенный.
- Прогресс и фазы видны пользователю (phase: "loading" / "parsing" / "building_graph" / "reachability").
- Нет длинных HTTP-запросов → дружит с прокси/cloud-таймаутами.
- Можно продолжать анализ после reload вкладки (запрашиваем `/status` по `project_id` из `localStorage`).

### Минусы и риски
- Сервер stateful → усложняет горизонтальное масштабирование (для MVP неактуально — запуск локальный).
- Память растёт с количеством активных проектов; нужен TTL и верхний лимит N одновременных проектов.
- Перезапуск сервера теряет проекты → клиент должен graceful-handle `404 project_not_found`.
- Дополнительный код: job registry, TTL GC, lock-контроль реанализа одного проекта.

### Сложность освоения: **Medium**

### Соответствие требованиям
- Закрывает все FR-01..26 и NFR-01..14 с запасом.
- **Лучше всего соответствует FR-07** (быстрая итерация entry points) и NFR-09 (UI живучесть — всё асинхронно, ошибки ловятся централизованно).
- Поддержка NFR-10 через `localStorage` + `project_id`.

## Вариант C — «SSE Streaming + Disk-cached Artifacts»

### Тех-стек
- Backend: Go stdlib + SSE (`text/event-stream`), disk-cache распакованных проектов и посчитанных графов в `${TMPDIR}/go-viz/<project_id>/...`.
- Frontend: React + Cytoscape.js + `EventSource` API, инкрементальная отрисовка по мере прихода кусков графа.
- Build/tooling: тот же Vite + `go build` + простой disk-GC (sweeper-горутина по TTL).
- Delivery: одиночный бинарь + Docker (+ volume для кэша, опционально).

### Ключевые идеи
- `POST /api/projects/{id}/analyze` → держит SSE-соединение, шлёт события: `phase`, `progress`, `warning`, `partial_graph` (кусками по N узлов), `done`.
- Распакованные файлы проекта и сериализованные `packages` кладутся на диск → переживают рестарт сервера.
- Re-analyze с новыми entry points переиспользует disk-кэш парсинга.
- Отчёты о мёртвом коде тоже генерятся стримингом.

### Плюсы
- Максимально живой UX: прогресс в реальном времени, первые узлы появляются до завершения анализа.
- Выживает рестарт сервера (disk cache).
- Меньше polling-трафика.
- Хорошо ложится на большие проекты (FR-18, агрегация поверх стрима).

### Минусы и риски
- Существенно больше кода: SSE-протокол, reconnect, backpressure, форматы чанков.
- Disk-cache → ответственность за инвалидацию, race-conditions, права доступа к tmp.
- Фронтенд сложнее: инкрементальный рендер Cytoscape при дельтах, race между `partial_graph` и пользовательскими действиями.
- EventSource не поддерживает кастомные headers и POST → часть API уедет в GET с query-params или потребует polyfill.
- Риск «перепроектировать MVP» и не уложиться в сроки защиты (до 15.05.26).

### Сложность освоения: **High**

### Соответствие требованиям
- Закрывает все FR/NFR, включая FR-18 особенно красиво.
- **Избыточность для MVP:** требования NFR-01..03 уже закрываются вариантом B без SSE; SSE даёт «плюс к UX», но ценой недели-двух разработки.
- Риск срыва сроков — конфликт с ограничением «один исполнитель, 26.04–15.05.26».

## Моя рекомендация (не решение)

**Вариант B.** Он:
- закрывает все FR/NFR без «формальных натяжек»;
- даёт реально хороший UX для FR-07 (главная фича — итеративные entry points);
- не создаёт риска срыва сроков;
- расширяется до C без выбрасывания кода (polling → SSE — прямолинейная миграция эндпоинта `status`).

Вариант A — если приоритет «минимум кода и идеальная простота деплоя» важнее UX итеративного анализа. Вариант C — если планируется развивать проект после защиты и хочется вложиться в инфраструктуру.

## Вопрос пользователю

**Какой вариант берём?**
Можно смешивать оси — например, «B, но без TTL-GC (вручную удалять /api/projects/{id})» или «B, но с disk-cache только для парсинга, без графов в памяти». Скажи конкретно, что меняешь.
